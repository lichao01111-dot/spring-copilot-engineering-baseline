# Spring 服务安全建模与控制标准

安全审查不应只搜索“SQL 注入”。每个新端点、消息消费者、文件/URL 处理、管理操作和外部集成都应建立以下威胁模型。

## 最小威胁表

| 资产/入口 | 攻击面 | 必须确认的控制 | 验证方式 |
|---|---|---|---|
| HTTP API | 未认证、越权、参数篡改、批量枚举 | 认证、资源级授权、输入校验、限流、审计 | Web/集成测试与审查 |
| 管理接口 | 权限提升、调试信息泄露 | 独立角色、最小权限、操作审计、网络隔离 | 配置/部署检查 |
| 文件上传/下载 | 路径穿越、恶意类型、资源耗尽 | 大小/类型限制、对象存储隔离、病毒扫描、授权下载 | 集成测试 |
| URL/回调 | SSRF、开放重定向、凭据泄露 | host allowlist、私网/metadata 阻断、超时、重定向限制 | 单元/集成测试 |
| 消息 | 伪造、重复、重放、敏感数据泄露 | 生产者信任、幂等、schema、最小 payload、DLQ 权限 | 容器测试与审查 |
| 数据库 | 越租户、注入、备份泄露 | 参数化、租户过滤、加密/脱敏、审计、备份访问控制 | Repository/运维检查 |

## 鉴权标准

1. 身份认证只能回答“是谁”；授权必须回答“能否操作此资源”。
2. 资源归属、组织/租户边界和管理角色必须在服务/用例层复核，不能只依赖前端传来的 tenantId。
3. 默认拒绝；新增端点必须显式声明访问策略。匿名访问只能按白名单开放。
4. 权限失败、资源不存在和租户隔离的响应策略应一致，避免资源枚举。

## 密钥和日志

密钥只来自受管注入渠道；不进入源码、镜像层、测试 fixture、错误响应或普通日志。日志采用字段级脱敏而不是依赖人工“不要打印”。异常链中可能含请求体、SQL 参数或远程响应时，必须在统一边界处理。
