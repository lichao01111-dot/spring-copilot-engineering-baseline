# 服务边界与单 Deployment 规则

## 默认规则

同一**业务聚合/业务能力**的完整生命周期必须归属同一个 Spring 服务：创建、查询、修改、删除/归档、状态转换、资源级授权、审计和对外 API 由同一服务拥有，并默认由一个 Kubernetes `Deployment` 发布。

这里的“同一”不是 URL 前缀相同，也不是数据库表相同，而是满足以下多数条件：共享业务不变量；需要本地事务保证一致性；由同一团队负责；共享数据所有权、权限模型、SLO 和发布节奏。

```text
订单聚合：创建订单 / 查询订单 / 修改草稿 / 取消订单 / 确认订单
    → order-service
    → 一个 order-api Deployment（可有多个 Pod 副本）
    → 该服务拥有订单数据与迁移
```

## 为什么这是合理的默认值

1. **一致性清晰**：写操作和读取语义能围绕同一个聚合、事务、版本号和权限模型设计。
2. **数据所有权清晰**：不会出现 A 服务写表、B 服务删表、C 服务读表的共享数据库耦合。
3. **降低分布式复杂度**：避免为了 CRUD 把一次本地状态变化拆成远程调用、事件补偿和跨服务排障。
4. **发布与回退更安全**：API、代码、Schema 和权限策略的兼容窗口由一个团队、一个发布单元管理。
5. **对 Agent 更可控**：Agent 能在明确边界内读取完整上下文、测试完整流程，而不是随意跨服务修改。

## 重要澄清

- 一个 `Deployment` 可以有多个 Pod 副本；“一个 Deployment”不是“单实例”。
- 代码内部可以采用 Command/Query 分层、独立读模型、缓存或只读副本；这不等于拆成两个服务。
- 逻辑服务在蓝绿、金丝雀、灾备或滚动发布期间可能短暂存在多个 Deployment；这属于部署策略例外，不是服务边界改变。
- `CronJob`、异步 worker 和数据回填任务如与 API 的资源/扩缩容模型不同，可以成为同一服务代码库下的独立工作负载，但必须记录例外 ADR、消费语义与部署所有权。

## 禁止的拆分方式

- 按 CRUD 拆成 `create-service`、`query-service`、`update-service`、`delete-service`。
- 按 Controller、表、前端页面或单个接口拆分服务。
- 一个服务拥有写权限，另一个服务绕过它直接查询或修改同一业务表。
- 仅因为“未来可能扩容”就提前拆出读写服务；先使用分页、索引、缓存、读副本和异步投影验证瓶颈。

## 允许拆分的证据门槛

拆分并不是禁止，而是必须由 `spring-architecture` 产出 ADR，且至少满足一项强证据：

| 拆分理由 | 必须提供的证据 | 拆分后必须定义 |
|---|---|---|
| 独立扩缩容 | 真实压测/生产指标显示读写资源模型显著不同，单 Deployment 无法经济满足 SLO | 读模型同步、延迟、过期数据语义、降级 |
| 安全/合规隔离 | 不同网络区、数据分级、访问主体或审计义务无法在同一工作负载满足 | 权限边界、数据流、密钥和审计责任 |
| 独立业务所有权 | 不同稳定团队对不同聚合拥有独立路线图、值班与发布权 | API/事件契约、数据迁移、责任人 |
| 独立可靠性目标 | 一个能力的故障域、可用性或延迟预算必须与另一个隔离 | 依赖、限流、熔断、回退、SLO |
| 独立运行模型 | API 与消费者/批处理的 CPU、内存、并发、生命周期显著不同 | 同代码库/独立镜像决策、队列、幂等、重试、部署 |

“团队想拆微服务”“技术上可以拆”“接口很多”都不是充分证据。

## 服务所有权不变量

每个服务边界必须记录：聚合及状态机、对外命令和查询、数据 Schema 所有权、权限主体、事件生产/消费、Deployment 名称、负责人、SLO、发布与回退责任。跨服务只能通过公开 API 或版本化事件通信，不能共享 Repository、Entity 或业务表写权限。

在多微服务 workspace 中，这些事实必须落到根 `workspace-map.md` 和每个服务的 `SERVICE.md`。Agent 可以使用 workspace 获得全局视图，但不能因为“能看到多个服务代码”就扩大服务边界或绕过公开契约。
