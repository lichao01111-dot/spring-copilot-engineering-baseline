# 安全与性能完成门禁

## 适用原则

“代码能编译、测试通过”不足以说明可以完成。任何新增端点、消息消费者、外部调用、数据访问、异步任务、缓存、导出、文件/URL 处理、认证授权、迁移和依赖升级，都必须在完成声明中给出与风险相称的安全和性能证据。

## 安全完成门禁

| 变更 | 必须证明 | 最低验证 |
|---|---|---|
| 新增/修改 API | 认证、资源级授权、输入校验、错误脱敏、限流策略 | 正常/无身份/越权/非法输入 Web 或集成测试 |
| 数据访问 | 租户/组织过滤、参数化、最小字段、审计 | Repository/服务层测试，审查查询 |
| URL、回调、文件 | SSRF/路径/大小/类型/下载授权控制 | allowlist、私网阻断、超时、负例测试 |
| 外部依赖 | TLS、超时、重试上限、凭据来源、失败降级 | 配置与集成测试/MockServer |
| 消息/异步 | 生产者可信、最小 payload、幂等、重放与 DLQ 权限 | 重复/失败/重试测试 |
| 依赖升级 | CVE、许可证、受影响调用面 | 扫描结果和升级验证 |
| 高风险 API | 业务必要性、替代方案、工单、负责人、测试和监控 | `security_preflight.py` 通过或正式例外 |

安全例外只能用于已有替代控制且无法避免的情况。注释格式为：

```java
// baseline-risk-allow: SEC-CMD-EXEC PLATFORM-1234
// 原因：调用公司受管的只读诊断二进制；参数来自固定枚举，进程权限最小化。
Process process = new ProcessBuilder(command).start();
```

例外必须关联工单和安全评估；“赶工”“历史代码”“本地能用”不是理由。例外注释不替代 Code Owner 审核。

## 性能完成门禁

| 变更 | 必须证明 | 最低验证 |
|---|---|---|
| 列表/搜索/导出 | 分页、最大值、稳定排序、索引和超时 | 大小边界测试；必要时执行计划 |
| 写入/状态转换 | 事务范围、锁/版本、幂等、重试预算 | 并发/重复测试 |
| DB 访问 | 不存在 N+1、无界加载、错误 fetch join 分页 | 集成测试、SQL/执行计划审查 |
| 外部调用 | connect/read timeout、连接池、熔断/降级、调用预算 | 超时和依赖故障测试 |
| 消息/批处理 | 批次、并发、背压、可续跑、DLQ | 积压/重复/失败恢复测试 |
| 缓存 | key、TTL、失效、一致性和容量 | 命中/过期/回源负载测试 |
| 资源敏感路径 | 延迟预算、并发、内存/CPU、线程/连接池 | 负载或基准测试，记录方法和结果 |

性能例外也必须引用工单：

```java
// baseline-risk-allow: PERF-UNBOUNDED-FIND-ALL REPORT-238
// 原因：该表由迁移保证最多 20 条系统配置；审查见 ADR-014。
return configurationRepository.findAll();
```

本地 `scripts/verify.sh` 默认以警告方式报告性能风险，避免遗留系统一次性无法迁移；CI 设置 `PERFORMANCE_RISK_MODE=strict`，新风险必须修复或有已批准例外。高风险安全用法本地和 CI 都默认阻断。

## 机械检查的边界

`security_preflight.py` 和 `performance_preflight.py` 只覆盖少量高价值模式，目的在于让风险显式，而非声称已完成 SAST、渗透测试或压测。企业项目仍应接入批准的 SAST、依赖扫描、镜像扫描、DAST、性能压测和运行时监控。扫描通过不等于安全或高性能；缺少威胁模型、性能预算或相关测试时，审查仍应失败。
