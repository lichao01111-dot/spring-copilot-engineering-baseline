# 工程度量与隐私治理

## 为什么采集

度量用于改进基线，而不是监控个人。要回答的问题包括：哪些 Guide 被使用、哪些检查经常失败、修复/审查的阻塞点在哪里、安全或性能风险是否集中、规则是否正在制造误报和额外等待。

禁止用度量进行个人排名、工时考核、提示词监控、源码分析或行为画像。

## 允许收集的字段

| 字段 | 用途 |
|---|---|
| `project_key` | 匿名项目维度；使用内部别名，不使用仓库 URL、客户名或服务真实名称 |
| `event_type`、`name` | Guide、Agent、检查或风险扫描的使用情况 |
| `outcome`、`reason_code`、`severity` | 失败/阻塞聚类与规则改进 |
| `duration_ms` | 粗粒度流程瓶颈；不等同于个人工时 |
| `environment` | 区分本地与 CI |

## 明确禁止

不得采集源码、diff、文件路径、提示词、用户消息、分支名、PR/Issue 链接、工单号、用户名、邮箱、IP、客户/租户标识、密钥、业务输入输出、堆栈、日志或原始测试结果。

`baseline-risk-allow` 中的工单号只能留在代码审查与例外流程中，不能写入度量事件。

## 存储、导出与保留

- 默认只写入本地 `.baseline/metrics/events.jsonl`，该文件被 `.gitignore` 排除。
- 默认不上传。`export_metrics.py --upload` 只有在配置 `allow_upload=true`、内部 HTTPS `endpoint` 后才会访问网络。
- 导出只发送聚合报告，不发送原始事件。
- 建议原始事件保留 90 天，聚合趋势保留 12 个月；具体以公司隐私、法务和数据保留政策为准。
- 端点鉴权使用 CI Secret 或 OIDC；不得在 `config.json`、代码或日志保存 Token。

## 指标解释原则

- 使用量低，不等于 Guide 无价值；先确认是否被正确安装、是否有替代流程。
- 失败率高，先分辨是产品需求、工具缺陷、遗留债务还是规则误报。
- 耗时上升，只能提示流程瓶颈，不能直接归因于个人能力。
- 修改规则前，应查看多个项目和多个周期的聚合证据，并小范围试点。
